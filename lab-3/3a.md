## Architecture Summary

```
User (anywhere)
   ↓
CloudFront (global edge) — WAF + Origin Cloaking
   ↓
Route 53 Latency Routing
   ↓                        ↓
Tokyo ALB              São Paulo ALB
shinjuku-alb01         liberdade-alb01
   ↓                        ↓
Tokyo EC2              São Paulo EC2 (stateless)
   ↓                        ↓
Tokyo RDS  ←————TGW Peering Corridor————
(PHI stored here only)
```

**One rule: Global access. No global storage. All PHI stays in Japan.**

## Apply Order (Critical)

TGW peering is a handshake — Tokyo requests, São Paulo accepts. Order matters.

### Phase 1 — Tokyo First Apply
```bash
cd tokyo/
terraform init
terraform apply
```
Creates: VPC, RDS, EC2, ALB, TGW, Secrets Manager locker, CloudFront, WAF, CloudTrail.

Comment out before this apply:
- `aws_ec2_transit_gateway_peering_attachment.shinjuku_to_liberdade_peer01` in `tokyo_main.tf`
- `aws_ec2_transit_gateway_route.tokyo_to_saopaulo` in `tokyo_tgw.tf`
- `aws_route.shinjuku_to_sp_route01` in `tokyo_main.tf`
- `output "tokyo_tgw_peering_attachment_id"` in `tokyo_outputs.tf`
- `aws_route53_record.origin_saopaulo` in `tokyo_acm.tf`

### Phase 2 — São Paulo First Apply
```bash
cd saopaulo/
terraform init
terraform apply
```
Creates: VPC, EC2, ALB, TGW, VPC attachment. Reads secret from Tokyo Secrets Manager via `aws.tokyo` provider.

Comment out before this apply:
- `aws_ec2_transit_gateway_peering_attachment_accepter.liberdade_accept_peer01` in `saopaulo_main.tf`
- `aws_ec2_transit_gateway_route.saopaulo_to_tokyo` in `saopaulo_main.tf`
- `aws_route.liberdade_to_tokyo_route01` in `saopaulo_main.tf`

Grab from outputs:
```
liberdade_tgw_id        → update var.saopaulo_tgw_id in tokyo_variables.tf
liberdade_alb_dns_name  → update var.saopaulo_alb_dns_name in tokyo variables.tf
```

### Phase 3 — Tokyo Second Apply (Peering Request)
Uncomment `shinjuku_to_liberdade_peer01` in `tokyo_main.tf` and `tokyo_outputs.tf`.
Keep TGW route and return route still commented out.

```bash

terraform apply
```

Immediately check the peering attachment ID — peering requests expire:
```bash

  --region ap-northeast-1 \
```

Update `saopaulo_variables.tf`:
```hcl

  default = "<attachment-id-from-above>"
}
```

### Phase 4 — São Paulo Second Apply (Accept Peering)
Uncomment in `saopaulo_main.tf`:
- `liberdade_accept_peer01`
- `saopaulo_to_tokyo` TGW route
- `liberdade_to_tokyo_route01`

```bash
cd saopaulo/
terraform apply
```

**Do this immediately after Phase 3 — peering requests expire.**

### Phase 5 — Tokyo Final Apply (Routes + DNS)
Uncomment in `tokyo_tgw.tf`:
- `aws_ec2_transit_gateway_route.tokyo_to_saopaulo`

Uncomment in `tokyo_main.tf`:
- `aws_route.shinjuku_to_sp_route01`

Uncomment in `tokyo_acm.tf`:
- `aws_route53_record.origin_saopaulo`

```bash
cd tokyo/
terraform apply
```

## Known Gotchas

### Route53 record already exists
Add `allow_overwrite = true` to the `aws_route53_record` resource in `cloudfront.tf`.

### Peering attachment in invalid state
Peering requests expire if not accepted quickly. If the attachment shows `deleted`, comment it out, apply Tokyo to remove it, then start Phase 3 again.

### TGW route fails with IncorrectState
The route cannot be created while the peering attachment is still `pendingAcceptance`. São Paulo must accept first. Keep the TGW route commented out until after Phase 4.

### Secrets Manager secret mismatch
If CloudFront and the ALB have different secret values, fix by syncing Secrets Manager to match what CloudFront has:
```bash
aws secretsmanager put-secret-value \
  --region ap-northeast-1 \
  --secret-id lab3/cloudfront/origin-secret \
  --secret-string "<value-cloudfront-is-sending>"
```
Then run `terraform apply` in Tokyo to update the ALB listener rule.

To prevent future drift, add to `tokyo_origin_secret.tf`:
```hcl
lifecycle {
  ignore_changes = [secret_string]
}
```

### São Paulo can't find Secrets Manager secret
São Paulo reads the secret from Tokyo's region. The `sao_paulo_data.tf` data sources must use `provider = aws.tokyo`, not `provider = aws.saopaulo`. The `saopaulo_provider.tf` must define an `aws.tokyo` alias pointing to `ap-northeast-1`.

### CNAMEAlreadyExists on CloudFront replace
Never use `terraform apply -replace` on a CloudFront distribution that owns a CNAME. The old distribution still holds the alias. Fix the secret sync issue directly instead.

```bash
aws ec2 describe-route-tables \
  --region ap-northeast-1 \
  --filters Name=vpc-id,Values=vpc-05340bd48477eff9a \
  --query "RouteTables[].Routes[?DestinationCidrBlock=='10.190.0.0/16']"
```
Expected: route with `State: active` via TGW.

### Confirm São Paulo route table has Tokyo CIDR
```bash
aws ec2 describe-route-tables \
  --region sa-east-1 \
  --filters Name=vpc-id,Values=vpc-04dc19eed15db2858 \
  --query "RouteTables[].Routes[?DestinationCidrBlock=='10.52.0.0/16']"
```
Expected: route with `State: active` via TGW.

### Test CloudFront cache miss then hit
```bash
curl -I https://thedawgs2025.click/api/public-feed   # X-Cache: Miss
curl -I https://thedawgs2025.click/api/public-feed   # X-Cache: Hit
```
Second response should include `Age: <seconds>`.

### Test São Paulo EC2 → Tokyo RDS connectivity
```bash
aws ssm start-session --target i-0083ffc43c15b06d3 --region sa-east-1
# Inside the instance:
nc -zv shinjuku-rds01.c76yg4640twf.ap-northeast-1.rds.amazonaws.com 3306
```
Expected: `Connected to 10.52.102.98:3306`

### Verify no RDS in São Paulo
```bash
aws rds describe-db-instances \
  --region sa-east-1 \
  --query 'DBInstances[].DBInstanceIdentifier'
```
Expected: `[]`

### Check CloudFront origin header matches ALB listener rule
```bash
# What CloudFront is sending:
aws cloudfront get-distribution --id EQUDJ3PPRYIZ \
  --query "Distribution.DistributionConfig.Origins.Items[*].CustomHeaders.Items"

# What the ALB is expecting:
aws elbv2 describe-rules --region ap-northeast-1 \
  --listener-arn $(aws elbv2 describe-listeners \
    --region ap-northeast-1 \
    --load-balancer-arn $(aws elbv2 describe-load-balancers \
      --region ap-northeast-1 --names shinjuku-alb01 \
      --query "LoadBalancers[0].LoadBalancerArn" --output text) \
    --query "Listeners[0].ListenerArn" --output text) \
  --query "Rules[?Priority=='1'].Conditions"
```
Both `HeaderValue` fields must match exactly.
